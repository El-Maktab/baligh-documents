\newcommand{\AbstractPage}{%
\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}
Arabic digital writing still suffers from a shortage of integrated tools that can correct Modern Standard Arabic text accurately, respond in real time, and explain their feedback in a way that helps users learn. Existing tools often focus on surface spelling fixes or black-box suggestions without exposing the grammatical reasoning behind the correction. Baligh addresses this problem by providing an Arabic writing assistant that combines grammatical error detection, grammatical error correction, spelling-aware editing, next-word suggestion, and explanatory feedback in a single user-facing system. The project aims to support students, educators, and Arabic content writers by improving writing quality while also making the correction process more understandable and educational.

The implemented system follows a modular pipeline. A preprocessing layer performs normalization, word-boundary handling, segmentation, and morphological analysis and disambiguation. On top of that, Baligh integrates a grammatical error detection engine that fuses rule-based, lexicon-based, and machine-learning detectors, then passes its findings to correction components built around ontology-guided reasoning, dictionary lookup, and text-editing models. In parallel, a next-word suggestion module supports both next-word prediction and word auto-completion. The final system highlights issues, proposes corrections, returns ranked suggestions, and presents grammar-oriented explanations to the user in an interactive workflow.

The project outputs include the preprocessing pipeline, grammatical error detection and correction services, the next-word suggestion subsystem, and a working user application. Testing and validation relied on automated unit and integration tests together with evaluation on established Arabic linguistic resources, including QALB14, QALB15, and ZAEBUC. Overall, Baligh demonstrates that a modular and explainable Arabic writing assistant can combine linguistic processing and machine learning into a practical platform for improving Arabic writing quality.
\clearpage
}

\section{System Architecture Context}
To understand where my modules fit, it is important to look at the high-level system architecture of Baligh. 


Data enters the system through the \textbf{Client Layer}, which consists of the user interface (web text editor) and the REST API backend. The raw text is immediately sent to the \textbf{Preprocessing Pipeline}. This pipeline performs foundational text cleaning, normalization, tokenization via Farasa, and contextual morphological analysis and disambiguation using CAMeL tools. 

Once the text is preprocessed, it is dispatched into the \textbf{Core Features Layer}, which splits into two parallel tracks:
\begin{enumerate}
    \item \textbf{The Slow Path (GEC \& GED):} Preprocessed text is fed into the Grammatical Error Detection (GED) Engine. These findings are passed to the Grammatical Error Correction (GEC) Module, which proposes corrections via an ontology-guided grammar lookup, dictionary spelling checks, and a text-editing machine learning model (the Edit Tagger). Each module generates human-readable explanations. A final fusion and ranking layer (the Ranker) consolidates the corrections and explanations.
    \item \textbf{The Fast Path (NWS):} For predictive typing, the Next-Word Suggestion (NWS) module operates with minimal latency.
\end{enumerate}

\section{Contribution 1: GEC Edit Tagger (Text Editing Sub-module)}
\subsection{Functional Overview}
The system implements a Grammatical Error Correction (GEC) model for Arabic using a \textbf{text editing (sequence tagging)} approach rather than a traditional sequence-to-sequence (Seq2Seq) generative model. By treating correction as a sequence of edit operations rather than generating text from scratch.

\textbf{Correction Generation}\\
Instead of generating the corrected sentence autoregressively, the model predicts a sequence of edit operations for each token in the erroneous input. These predicted edits are then applied to the original text to produce the corrected sentence.

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{assets/tag_approach.png}
\caption{Overview of the sequence-tagging text editing approach.}
\label{fig:tag_approach}
\end{figure}

\begin{itemize}
    \item \textbf{Edit Extraction \& Alignment:} The system aligns erroneous and corrected sentence pairs using a weighted Levenshtein edit distance, accompanied by an iterative algorithm to capture multi-word errors. It then applies the character-level alignments.
    \item \textbf{Applying Operations:} It assigns specific character-level edits to the input words to transform them. The primary operations include:
    \begin{itemize}
        \item \textbf{Keep (K)}
        \item \textbf{Delete (D)}
        \item \textbf{Insert (I\_[])}
        \item \textbf{Replace (R\_[])}
    \end{itemize}
    \item \textbf{Optimization Strategies:} To keep the vocabulary of possible edits manageable and improve learning, the system uses:
    \begin{itemize}
        \item \textbf{Edit Compression:} Merging consecutive operations (e.g., consecutive deletes \texttt{DDD} into \texttt{D*} or insertions into \texttt{I\_[تت]} \texttt{I\_[ت*]}).
        \item \textbf{Sub-word Projection:} Edit operations are first extracted at the word level, where grammatical corrections are naturally defined. These edits are then projected onto the corresponding sub-word tokens so that each token receives a label, matching the tokenization scheme used by Transformer models such as BERT.
        \item \textbf{Edit Segregation:} Splitting the correction into a two-step pipeline—one model for non-punctuation errors, and a subsequent model for punctuation errors.
        \item \textbf{Edit Pruning:} Removing rare edit operations during training to help the model focus on frequent and informative edits.
    \end{itemize}
\end{itemize}

\textbf{Explanation Generation (Interpretability)}\\
Unlike autoregressive Seq2Seq models, which generate only the final corrected sentence, edit-based models explicitly predict the edit operation applied to each input token (e.g., \textbf{KEEP}, \textbf{DELETE}, \textbf{REPLACE}, or \textbf{INSERT}). These edit labels provide a transparent record of how the input was transformed, making it straightforward to identify the exact grammatical changes performed. This improves the interpretability of the correction process and facilitates error analysis, debugging, and the generation of user-friendly explanations for the applied corrections.

\subsection{Modular Decomposition}
\begin{itemize}
    \item \textbf{Alignment Module}\\
    The aligner is responsible for aligning the erroneous sentence with its corrected one using the Levenshtein edit distance algorithm. The alignment is performed in two stages. First, alignment is carried out at the \textbf{word level} to determine the correspondence between words in the two sentences. At this stage, operations such as \textbf{insertions}, \textbf{deletions}, \textbf{merges}, and \textbf{splits} are identified to handle cases where a single word in one sentence corresponds to multiple words in the other, or vice versa. Once the word-level alignment has been established, each aligned word pair is further aligned at the \textbf{character level} to determine the exact character insertions, deletions, and replacements required to transform the erroneous word into its corrected form. This hierarchical alignment provides the detailed edit information that serves as the basis for extracting edit labels used during model training.
    
    \item \textbf{Rewrite Module}\\
    After the initial alignment, the rewriter generates an intermediate representation of the aligned sentences. This step resolves ambiguities in the character-level alignment and rewrites complex edits into a consistent form that can be more easily projected onto tokens.
    
    \item \textbf{Edit Projection Module}\\
    Since grammatical edits are initially extracted at the character level while Transformer-based models operate on tokenized input, the projector maps the character-level edit operations onto the corresponding tokens or sub-word tokens. Each token is assigned an edit label describing the operation that should be applied, making the extracted edits compatible with the model's token-level prediction mechanism.
    
    \item \textbf{Edit Compression Module}\\
    The projector may produce multiple consecutive edit operations for a single token. The compressor combines these individual operations into a compact edit label whenever possible. This reduces redundancy in the label sequence, simplifies the classification problem, and produces a more manageable set of edit tags for model training and inference.
    
    \item \textbf{Edit Pruning Module}\\
    The pruner analyses the frequency of all generated edit labels across the training corpus and removes edit labels that occur below a predefined frequency threshold. Rare edit operations contribute little to the learning process while significantly increasing the number of output classes. Pruning these infrequent labels reduces label sparsity, decreases vocabulary size, and allows the model to focus on learning the most common and informative correction patterns.
    
    \item \textbf{Edit Segregation Module}\\
    Punctuation corrections often exhibit different characteristics from grammatical and spelling corrections. The segregator separates punctuation-related edit labels from non-punctuation edits, allowing them to be processed independently. This separation improves the handling of punctuation corrections and enables the system to treat punctuation edits differently during training or inference if required.
    
    \item \textbf{Edit Application Module}\\
    During inference, the model predicts a sequence of edit labels rather than generating corrected text directly. The edits applicator interprets these predicted labels and applies the corresponding operations—such as insertions, deletions, replacements, merges, or splits—to the original input tokens. The resulting sequence of modified tokens is then reconstructed to produce the final corrected sentence.
\end{itemize}

\subsection{Design Constraints}
\begin{itemize}
    \item The edit tagging task exhibits a significant class imbalance, as the majority of tokens in the training corpus do not require correction and are therefore assigned the KEEP label. This imbalance may bias the model toward predicting KEEP more frequently than edit operations, potentially reducing its ability to detect less common grammatical errors.
    \item The frequency of grammatical error types in the training corpus is highly uneven. Errors related to Hamza (\foreignlanguage{arabic}{ء,أ,إ}), Taa' Marbuta (\foreignlanguage{arabic}{ة,ه}), and punctuation constitute a large proportion of the annotated corrections, whereas many other grammatical phenomena occur much less frequently. \\
    These corpus characteristics directly influence the model's learning behavior. Frequent edit operations are learned more reliably due to their abundance in the training data, while infrequent grammatical phenomena remain more challenging to detect and correct.
\end{itemize}

\section{Contribution 2: Ranker Sub-module}

\subsection{Functional Description}
The Ranker module is responsible for producing the final coherent text alongside its supporting grammatical explanations.

\begin{itemize}
    \item \textbf{Correction Generation} evaluates the corrections coming from GEC making use of the GED output. It applies a rigorous filtering and scoring pipeline to candidates from multiple modules. Overlapping or conflicting candidate edits are resolved by selecting the candidate with the highest composite score.
    \item \textbf{Explanation Generation} is facilitated by hardcoded scoring biases that favour explainable, rule-based corrections. Specifically, if a detected error is marked as \texttt{explanation\_eligible} and the candidate correction comes from the rule-based \texttt{ONTOLOGY} module, a positive weight (\texttt{W\_EXPLAIN}: +0.05) is applied. This bias guarantees that the final selected corrections can be mapped back to transparent linguistic rules rather than relying solely on black-box predictions.
\end{itemize}

\subsection{Modular Decomposition}
This module is decomposed into four primary submodules:

\begin{enumerate}
    \item \textbf{Configuration Manager}\\
    Manages the tunable hyper-parameters used during scoring. Key configurations include base module weights (\texttt{W\_ONTOLOGY}: 0.25, \texttt{W\_TAG}: 0.15, \texttt{W\_DICTIONARY}: 0.10), confidence weightings (\texttt{W\_EDIT\_CONF}: 0.30), and penalty weights (\texttt{W\_CHAR\_DIST}: 0.10, \texttt{W\_LOW\_CONF}: 0.15).
    
    \item \textbf{Candidate Aggregator \& Matcher}\\
    Maps candidate edits to specific errors detected by the GED by checking span intersections. Candidates whose correction equals the original text are immediately filtered out.
    
    \item \textbf{Scoring Engine}\\
    Computes a comprehensive \texttt{score} for each candidate edit using multiple weighted heuristics:
    \begin{itemize}
        \item \textbf{Base Module \& Confidence}: Adds base module weight, edit confidence (\texttt{W\_EDIT\_CONF}), and GED confidence (\texttt{W\_GED\_CONF}).
        \item \textbf{Provenance Tiers}: Boosts scores for \texttt{tier\_1\_rule\_derived} (+0.15) and \texttt{tier\_2\_rule\_supported} (+0.08) error spans.
        \item \textbf{Linguistic Alignment}: Awards +0.10 (\texttt{W\_SPELL\_DICT}) if a Dictionary module corrects an \texttt{ORTHOGRAPHY} error, and +0.10 (\texttt{W\_GRAM\_ONT}) if the Ontology module corrects a \texttt{SYNTAX} or \texttt{MORPHOLOGY} errors. Out-of-vocabulary (\texttt{is\_oov}) tokens receive a +0.05 bonus.
        \item \textbf{Multi-Source Agreement}: Awards +0.20 (\texttt{W\_AGREEMENT}) for each peer module that suggests the exact same correction string.
    \end{itemize}
    
    \item \textbf{Final Selection \& Metadata Generator}\\
    Sorts the scored candidates per error using a strict tuple key: \texttt{(score, edit\_confidence, module\_priority, -length\_of\_correction)}. \texttt{module\_priority} is strictly ranked: Ontology (3) > Tag (2) > Dictionary (1).
    It then iterates chronologically through the text, applying the best candidate. Finally, it emits the final correction.
\end{enumerate}

\subsection{Design Constraints}
\begin{itemize}
    \item \textbf{Grammatical Accuracy and Precision}:\\
    The system penalizes edits that diverge too much from the original text. A penalty (\texttt{W\_CHAR\_DIST}: -0.10) is applied proportionately to the normalized Levenshtein distance, and another penalty (\texttt{W\_LENGTH\_RATIO}: -0.05) is applied for deviations in string length. Furthermore, if a candidate's \texttt{edit\_confidence} falls below \texttt{CONF\_LOW\_THRESHOLD} (0.30), a flat penalty of -0.15 (\texttt{W\_LOW\_CONF}) is applied.
\end{itemize}

\section{Contribution 3: Backend Integration (GEC Wrapper \& Baligh Wrapper)}
Beyond developing individual analytical models, a significant portion of my work involved integrating the dispersed NLP components into a unified, high-performance backend. This was achieved through two critical orchestration layers: the GEC Wrapper and the Baligh Wrapper.

\subsection{The GEC Wrapper}
The GEC Wrapper serves as the unified interface for all correction submodules. Before this wrapper existed, the Ontology, Dictionary, and Edit Tagger modules operated in isolation. The wrapper encapsulates them, managing the asynchronous dispatch of preprocessed text and GED findings to each module concurrently. It collects the varying correction candidates, standardizes their formats into a uniform \texttt{CorrectionCandidate} schema, and feeds them directly into the Ranker. This abstraction guarantees that adding or removing future correction models will not break the downstream application logic.

\subsection{The Baligh Wrapper}
The Baligh Wrapper sits at the highest level of the Core Features Layer. It is the master orchestrator that receives requests directly from the REST API. Its responsibilities include:
\begin{itemize}
    \item \textbf{Pipeline Coordination}: Routing text through the Preprocessing Pipeline, then fanning out to the GED and NWS modules, and subsequently passing GED results into the GEC Wrapper.
    \item \textbf{Latency Management}: Utilizing Python's \texttt{asyncio} to ensure that heavy machine-learning inferences (like the Edit Tagger) do not block faster rule-based or dictionary lookups.
\end{itemize}

\chapter{Challenges, Lessons Learned, and Future Work}
\section{Faced Challenges}
One of the major challenges was dealing with Arabic's morphological richness during edit projection. Aligning character-level edits to sub-word tokens accurately required handling complex affixes and clitics seamlessly. Another significant challenge was latency tuning in the Baligh Wrapper. Balancing the heavy computational requirements of sequence-tagging models against the strict real-time constraints of a web editor required careful asynchronous design and payload minimization.

\section{Lessons Learned}
\begin{itemize}
    \item \textbf{Interpretability Matters}: Moving from generative Seq2Seq to an edit-tagging approach proved that making AI explainable is just as important as making it accurate. Users trust corrections more when they can see exactly what changed.
    \item \textbf{The Benefit of Good Communication and Documentation}: Clear interfaces between modules (e.g., the standardized \texttt{CorrectionCandidate} schema) prevented integration breakdowns across the team.
\end{itemize}

\section{Future Enhancements}
Future iterations could explore replacing the hard-coded heuristic Ranker with a lightweight machine-learning ranker trained specifically on user acceptance rates. Additionally, the GEC Wrapper could be optimized further with model quantization or ONNX runtime integration to reduce the memory footprint of the Edit Tagger, improving the overall throughput of the Baligh Wrapper. The edit vocabulary could also benefit from more aggressive compression strategies and better handling of underrepresented grammatical phenomena in the training corpus.
Training on More Diverse Corpora: The current model was trained on corpora in which certain error categories, such as Hamza, Taa' Marbuta, and punctuation mistakes, are significantly more frequent than others. As a result, the model is naturally more proficient at correcting these common errors. Future work should focus on training the model using larger and more diverse corpora containing richer grammatical phenomena, including complex syntactic, semantic, and stylistic errors.
Expanding the Edit Vocabulary: Some rare edit operations were removed during edit pruning to control the size of the output label space. Future work may explore adaptive or hierarchical edit representations that preserve rare corrections without substantially increasing model complexity.

\chapter{Edit Tagger Tagset Reference}
This appendix summarizes the core edit operations predicted by the Grammatical Error Correction Edit Tagger. While the full vocabulary of edits is dynamically extracted from the training corpora and can include thousands of specific \texttt{REPLACE} and \texttt{INSERT} strings, the underlying mechanisms rely on these fundamental operations.

\section{Core Editing Operations}
\begin{itemize}
    \item \textbf{KEEP (K)}: Indicates that the token is grammatically and orthographically correct and should remain unchanged in the final output. This is the most frequently predicted class due to class imbalance.
    \item \textbf{DELETE (D)}: Indicates that the token is extraneous or incorrect and should be completely removed from the sentence.
    \item \textbf{REPLACE (R\_[text])}: Instructs the applicator to substitute the current token with the provided string. This is commonly used for spelling corrections, I'rab fixes, and morphological adjustments. Example: \texttt{R\_[ال]} or \texttt{R\_[في]}.
    \item \textbf{INSERT (I\_[text])}: Instructs the applicator to insert a missing word or character adjacent to the current token. Example: \texttt{I\_[أن]}.
\end{itemize}

\section{Operation Compression}
To keep the vocabulary size manageable, consecutive identical operations are compressed during the extraction phase and decompressed during application.
\begin{itemize}
    \item \textbf{D*}: Represents multiple consecutive deletions.
    \item \textbf{I\_[text*]}: Represents complex consecutive insertions that are merged into a single tag.
\end{itemize}
